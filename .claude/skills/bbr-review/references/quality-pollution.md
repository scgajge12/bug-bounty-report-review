# 品質汚染チェックガイド（スキャナ垂れ流し / AI 生成丸投げ）

近年、HackerOne / Bugcrowd / Intigriti / YesWeHack はいずれも、自動スキャナの出力垂れ流しと AI 生成文の未検証提出を**スパム扱い**で取り締まっている。検出されればその場の Informative だけでなく、**Reputation スコアの永続的な棄損 → アカウント停止**のリスクがある。

このガイドは観点 13 の判定根拠として使う。検出指標を網羅し、検出時の対処と修正方針までを示す。

---

## 0. 2025-26 年の業界実態

### HackerOne 公式ポリシー（最新）
- **AI 利用そのものは禁止していない**（Code of Conduct）
- ただし「ハルシネーション、不正確な技術内容、低努力ノイズ」は**スパム判定** → エンフォースメント対象
- 2025 年から **Hai Triage**（半自動トリアージシステム）を運用：AI が重複検出と低努力レポートのフラグを行ってから人間トリアージへ
- 2025 年の有効提出数は約 85,000 件（前年比 +7%）だが、低品質 AI 生成レポートの急増でレビュー負荷が増大

### 業界事例（公開情報ベース）
- **Nextcloud**（2026 年 4 月）：HackerOne 経由の AI 生成レポート急増を理由にバグバウンティプログラムを終了
- **Curl**：Daniel Stenberg が「時間泥棒 AI レポート」との戦いを公言、AI 生成判定の対応プロセスを公開
- 「AI Slop」が業界用語化、各プラットフォームで対応強化中

### Intigriti 公式トリアージスタンダード（v1.3, 2026-01）
- 直接 AI に言及はないが、PoC 要件で「**最もシンプルな方法で攻撃の悪用可能性を疑いなく証明**する」を強調
- 自動スキャナ出力のみでは「疑いなく証明」基準を満たさない
- 検出時はスパム判定 → 警告 → アカウント制裁

### このスキルでの判断指針
- AI を「下書き支援」として使うのは正常
- AI を「丸投げで提出」するのは禁止
- スキャナを「Hint として使う」のは正常
- スキャナを「結果コピペ」するのは禁止
- **人間が書いた定型文（テンプレの使い回し）も同様に観点 13 の汚染対象**：Bugcrowd 公式指摘では 70% のレポートで Impact が汎用テンプレ。発生源（スキャナ・AI・他人のテンプレ・過去の自分のレポート）に関わらず、「対象アプリ非依存の文」は減点。
- 観点 13 のスコアは、**「人間による精査・対象アプリ固有化の痕跡」が見えるか**で判断する

## 1. スキャナ垂れ流しの検出指標

### 該当ツール
- Nuclei
- Nessus
- Acunetix / Burp Scanner / OWASP ZAP
- Nikto
- Wapiti
- 各種 SaaS スキャナ（Detectify、Probely 等）

### 「スキャナ垂れ流し」が NG な理由
- **誤検知 (False Positive)**：スキャナはバージョン番号やエラーメッセージなど「表面的なパターン」で判定するため、実際には攻撃不可能なケースが多々ある
- **文脈の欠如**：スキャナは「そこに脆弱性がある可能性」を示すが、「ビジネスへの影響」までは理解していない
- プログラム規約で多くが明示禁止（HackerOne ポリシー、Bugcrowd Researcher Code of Conduct 等）

### 検出サイン

#### サイン 1：バージョン情報や HTTP ヘッダのみを根拠にしている
```
❌ NG 例:
The server returns "Server: Apache/2.4.41" which has known
vulnerabilities CVE-2021-XXXX, CVE-2022-XXXX, CVE-2023-XXXX.
```
- バージョン情報があるだけでは「攻撃可能性」は立証されない
- 該当 CVE のエクスプロイトコードが**実際に通る**ことを手動再現していない

#### サイン 2：CVE / CWE 番号の羅列のみで PoC なし
```
❌ NG 例:
Vulnerabilities found:
- CVE-2023-12345 (CWE-79)
- CVE-2023-67890 (CWE-89)
- CVE-2024-11111 (CWE-22)
```
- 列挙だけで、各 CVE が「対象アプリで実際に発火する」検証がない

#### サイン 3：スキャナ特有の出力フォーマットが残っている
- `[INFO]`、`[CRITICAL]`、`[MEDIUM]`、`[+] Found:` のようなツール固有の出力
- Nuclei テンプレート ID（`http/cves/2023/CVE-2023-XXXXX.yaml` など）
- Nessus プラグイン ID（`Plugin ID 12345`）
- Burp Scanner の Issue Definition がそのまま貼られている

#### サイン 4：「手動再現の証拠」が欠如
- リクエスト・レスポンスのコピーがない
- 攻撃シナリオの 3 段（Attacker / Victim / System）が空白
- スクショ・動画なし

#### サイン 5：1 レポートに複数の無関係な脆弱性
- 「サブドメイン X の SSL 証明書期限切れ」「サブドメイン Y のディレクトリリスティング」「サブドメイン Z の CSP 不備」が 1 レポートに同居
- スキャナの一括スキャン結果をそのまま貼り付けたパターン

### 「スキャナ起点だが OK」のパターン
スキャナ自体は強力なツールであり、「**手動で攻撃的な再現ができている**」場合は問題ない。以下は OK：
- スキャナで Hint を得たが、手動でペイロードを構築し、リクエスト/レスポンスを取得し、攻撃シナリオを構築している
- スキャナ警告のうち 1 つを深掘りし、独立したレポートとして提出している

### 検出時の対処
ユーザーへ伝える例：
```markdown
### 13. 品質汚染チェック (3/10) — **重度**

このレポートはスキャナ垂れ流しの兆候があります：
- バージョン情報のみを根拠とし、CVE-XXXX に対するペイロード送信と
  レスポンス検証が欠けています
- `[CRITICAL]` などのスキャナ固有出力が残っています
- 手動再現の HTTP リクエスト/レスポンスがありません

**提出非推奨**：このまま提出すると HackerOne では Spam 判定される
可能性が高く、Reputation スコアの棄損リスクがあります。

**修正方針**：
1. CVE-XXXX に対する具体的なペイロード（PoC）を構築し、対象環境で
   実際に発火することを確認する。
2. リクエスト/レスポンスをテキストで貼り、Attacker → Victim → System
   の 3 段で攻撃シナリオを書き直す。
3. それができない場合（誤検知だった場合）は、提出しない。
```

## 2. AI 生成丸投げの検出指標

### 「AI 生成丸投げ」が NG な理由
- **ハルシネーション**：AI は自信たっぷりに嘘をつく
  - 存在しない CWE / CVE 番号
  - 実在しないパラメータ・ファイルパスを攻撃箇所として記述
  - 辻褄の合わない攻撃シナリオ
- プラットフォーム規制：HackerOne / Bugcrowd 等が AI 生成の不正確レポートを明示的に禁止 or 制限し始めている
- AI 任せでも最終責任は提出者にある

### 検出サイン

#### サイン 1：過度に流暢で対象非依存の汎用文
```
❌ AI 臭い文:
"This vulnerability poses a significant risk to the confidentiality,
integrity, and availability of the application and its users.
Malicious actors could potentially leverage this issue to compromise
sensitive information and disrupt normal operations."
```
特徴：
- 「significant」「potentially」「leverage」「compromise」「disrupt」など曖昧で重みのある単語の濫用
- CIA を3つ並べる定型表現
- 対象アプリ名・実機能名・実エンドポイントの言及ゼロ
- どの脆弱性レポートにもそのまま貼れる汎用性

#### サイン 2：存在しない CWE / CVE / ライブラリ名
- `CWE-9999` のように MITRE 公式リストに存在しない番号
- 形式が不自然な CVE（例：`CVE-2024-ABCDE`）
- 「`SafeSanitizer` ライブラリを使用してください」（実在しない）
- バージョン番号が現実離れ（`v45.2.1` 等）

#### サイン 3：辻褄の合わない攻撃シナリオ
- Self-XSS なのに「他のユーザーが影響を受ける」
- GET リクエストの脆弱性なのに「CSRF トークンを盗む」
- 認証必須機能を「未認証で攻撃可能」
- パストラバーサル PoC が「Windows の /etc/passwd を読み取る」

#### サイン 4：同じ意味の文を別表現で繰り返す
```
❌ AI 臭い:
The application fails to properly validate user input. As a result,
the input is not sanitized correctly. This lack of input validation
allows attackers to inject malicious payloads. The absence of
sanitization mechanisms enables the execution of harmful code.
```
- 中身は「入力値検証が無い」の 1 文だが 4 回反復している

#### サイン 5：関係のない一般論の混入
- 「セキュリティはチーム全体の責任です」
- 「定期的なセキュリティ監査が推奨されます」
- 「このようなインシデントを防ぐためにはセキュリティ意識が重要です」
- → バグバウンティレポートに必要ない一般論

#### サイン 6：不自然な箇条書きの連続
- 中身が薄い箇条書きが 5-7 項目並ぶ
- 各項目が抽象的で「対象アプリの何が違うのか」が読み取れない
- ChatGPT / Claude 系の典型的な出力パターン

#### サイン 7：対象アプリ名の言及が一切なく、攻撃説明だけがテンプレ的
- Description / Steps / Impact のいずれにも対象アプリの実エンドポイント・実機能名が出てこない
- 「the application」「the endpoint」「the user」のような汎用代名詞のみ

#### サイン 8：英語が完璧すぎる + 機械的な構造
- 接続詞・段落構造が教科書的に整いすぎ
- 自分の書く英語のクセと乖離
- 主語の選択が単調（"The vulnerability...", "The attacker...", "The system..." の繰り返し）

### 「AI 補助だが OK」のパターン
AI を「下書き支援・英文校正」として使うのは正常。以下は OK：
- AI で構成案を作り、自分で対象アプリ固有の情報を埋め込んだ
- AI で英文校正を受けたが、各セクションの事実関係は自分で検証している
- 攻撃シナリオ・PoC・Impact が対象アプリ固有のコンテキストで書かれている

### 検出時の対処
```markdown
### 13. 品質汚染チェック (4/10) — **中程度**

このレポートには AI 生成丸投げの兆候が見られます：
- Description が "This vulnerability poses a significant risk..." の
  汎用テンプレ文で始まっており、対象アプリ `app.example.com` の固有
  コンテキストが含まれていません
- Impact セクションで CIA を 3 つ並べる定型表現が 2 段落続きます
- 「`SafeInputValidator` ライブラリ」と推奨していますが、実在を
  確認できませんでした（AI ハルシネーションの可能性）

**修正方針**：
1. AI 出力を「下書き」として扱い、各セクションに **対象アプリの実
   エンドポイント・実機能名・実データ種別**を必ず含める
2. 推奨ライブラリは実在するもの（DOMPurify、Bleach、Helmet.js 等）
   から選ぶ
3. 攻撃シナリオが対象アプリで論理的に成立するか自分で 1 行ずつ検証
```

## 3. 観点 13 のスコアリング判定フローチャート

```
スキャナ垂れ流しの検出サインに該当？
├── Yes → サインの数で減点
│   ├── 1 つ → 7/10
│   ├── 2 つ → 4/10（中程度）
│   ├── 3 つ以上 → 2/10（重度、提出禁止）
│   └── 完全コピペ → 0/10
└── No → AI 生成丸投げの検出サインに該当？
    ├── Yes → サインの数で減点
    │   ├── 1-2 つ（軽度） → 7/10
    │   ├── 3-4 つ（中程度） → 4/10
    │   ├── 5 つ以上（重度） → 2/10（提出禁止）
    │   └── 完全 AI 生成 → 0/10
    └── No → 10/10（汚染なし）
```

## 4. 提出後にバレるとどうなるか

各プラットフォームの実例（公開ポリシー / コミュニティ報告ベース）：

- **HackerOne**：Signal スコア低下 + Hai Triage で事前フィルタ強化 → プライベートプログラム招待が来なくなる → 重ねるとアカウント停止
- **Bugcrowd**：Researcher Trust Score 低下 → 同様に招待減 → アカウント停止
- **Intigriti**：Researcher Reputation 低下、v1.3 トリアージスタンダードで「警告 → 制裁 → 停止」の段階的エスカレーション明文化 → 公開プログラムのみアクセス → 停止
- **YesWeHack**：Reputation 低下 → 同停止

一度棄損した Reputation は、新規アカウントで一からやり直しても**追跡される可能性が高い**（メールアドレス、IP、デバイスフィンガープリント、税務情報）。失う代償が極めて大きい。

**業界トレンド**：Nextcloud のようにプログラム自体が終了するケースが出てきている。低品質提出はあなた個人だけでなく、**業界全体のバグバウンティ機会を狭める**行為と認識する。

## 5. プロハンターのアクションフロー

スキャナを使う場合：
1. スキャナの結果はあくまで「手がかり（Hint）」
2. 警告された箇所に対し、実際にペイロードを送信
3. 脆弱性の発火を確認 → リクエスト/レスポンス取得
4. 攻撃シナリオを構築（Attacker → Victim → System）
5. ビジネスインパクトを対象アプリ固有で記述
6. それでも報告できる材料が揃ったら提出

AI を使う場合：
1. AI は「下書き構成」「英語校正」のみ
2. 各セクションに対象アプリ固有のコンテキストを必ず入れる
3. CWE / CVE / ライブラリ名は MITRE / NVD / 公式ドキュメントで実在確認
4. 攻撃シナリオを 1 行ずつ「実際に成立するか」自分で検証
5. AI が書いた文章を**自分の言葉で**最低 30% 書き換える
6. 提出前に「これ AI に書かせた感じが残ってないか」をセルフチェック

## 6. レビューでこの観点を出すときのトーン

- 攻撃的にならない：「品質汚染」と言われると人格攻撃に聞こえやすい。事実ベースで「検出サインに該当している」「修正方針はこれ」と機械的に伝える
- 善意の AI 補助も汚染と区別する：完全 AI 生成ではない場合、「下書き支援としては正常な使い方ですが、最終提出にはこの部分の対象アプリ固有化が必要」と建設的に
- 「次回からの対策」も併記：このレポートだけでなく、次のレポートで同じ失敗を繰り返さないための原則を 1-2 行で
