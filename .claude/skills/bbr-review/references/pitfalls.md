# 初心者バグハンターがやりがちな失敗パターンと回避法

レビューで遭遇率が高い失敗パターンを整理。レビュー時にこれらに該当していたら必ず指摘する。

**最重要 2 件**：項目 0a「スキャナ垂れ流し」と 0b「AI 生成丸投げ」は、検出されると一発でアカウント停止リスクに直結する。詳細は `quality-pollution.md` を参照。

---

## 0a. スキャナ結果の「コピペ報告」

### 症状
- Nuclei / Nessus / Burp Scanner / Acunetix の出力をそのままレポート化
- バージョン情報や HTTP ヘッダだけを根拠に「脆弱性」と主張
- 手動再現の証拠（リクエスト/レスポンス/影響）がない
- スキャナ固有の出力（`[CRITICAL]`、Plugin ID 等）が残ったまま

### なぜ NG
- 多くが誤検知（False Positive）
- 文脈の欠如（ビジネス影響が分からない）
- 各プラットフォームでスパム判定 → Reputation 棄損 → アカウント停止

### 回避法
- スキャナ結果は「手がかり」止まり
- 必ず手動で：(1) ペイロード送信、(2) 脆弱性発火確認、(3) リクエスト/レスポンス取得、(4) 攻撃シナリオ構築
- それができたものだけを 1 件ずつ独立レポートとして提出

---

## 0b. AI 生成テキストの「未検証提出」

### 症状
- 過度に流暢で対象非依存の汎用文（「This vulnerability poses a significant risk...」）
- 存在しない CWE / CVE / ライブラリ名（ハルシネーション）
- 辻褄の合わない攻撃シナリオ（自己 XSS で「他ユーザー攻撃」と矛盾）
- 同じ意味を別表現で 3-4 回繰り返す
- 対象アプリ名の言及が一切ない

### なぜ NG
- AI は自信たっぷりに嘘をつく
- 各プラットフォームで明示的禁止または制限が始まっている
- 提出した責任は AI ではなく**あなた**にある

### 回避法
- AI は「下書き構成 + 英文校正」のみに使う
- 各セクションに対象アプリ固有のコンテキスト（実エンドポイント、実機能名、実データ種別）を必ず追記
- CWE / CVE / ライブラリ名は MITRE / NVD / 公式ドキュメントで実在確認
- 提出前に「これ AI に書かせた感じが残っていないか」セルフチェック

## 1. 「テンプレに当てはめただけ」レポート

### 症状
- どの脆弱性でも書ける汎用文：「This vulnerability allows attackers to compromise the application」
- Impact セクションが対象アプリ非依存のコピペ
- Steps が抽象的（「Send the malicious payload」など）

### なぜ起きるか
有名ハンターのテンプレを使い回すと、対象アプリ固有の文脈が抜け落ちる。Bugcrowd は「70% 以上のレポートが定型 Impact」と指摘している。

### 回避法
レポート作成時に「この文章、別の対象アプリにそのまま使い回せるか？」と自問する。Yes なら書き直し。**対象アプリの実名・実エンドポイント・実データ種別を最低 1 箇所、Impact セクションに含める**。

### レビューでの指摘テンプレ
> Impact が汎用テンプレになっています。「any visitor」「sensitive data」のような抽象語ではなく、`example.com` のどの機能・どのデータ種別が侵害されるか、ユーザー数の規模（公開統計があれば）まで具体化してください。Bugcrowd の調査によれば、定型 Impact は Severity ダウングレードや Informative 判定の主要因です。

---

## 2. 「動かない PoC」を出す

### 症状
- ペイロードに改行が混入してコピペで動かない
- URL に余計な空白
- 認証必要なエンドポイントなのにクッキー指定がない
- 環境固有（自分のローカル DNS、自分のブラウザ拡張）でしか動かない

### 回避法
**シークレットウィンドウで PoC を再現してから提出**。Burp の Repeater で送り直して 200 が返ることを確認。`curl` コマンドで貼り付け可能な形に整える。

### レビューでの指摘
> PoC が再現可能か検証しましたか？ Cookie / 認証ヘッダ / Content-Type が抜けていないか確認してください。Triager は最初の 1 分で動かなかったら次の指摘質問を投げて差戻しになります。

## 2.5. 過剰な主張（Overclaiming）— 「専門知識がない」と即断される NG

### 症状
- Self-XSS を「Web サイト全体への攻撃が可能」と書く
- Reflected XSS を「Critical, full RCE」と書く
- Open Redirect 単体を「Critical, account takeover」と書く
- バージョン情報の開示だけで「Critical, RCE possible」と書く
- 認証必須機能の脆弱性を「未認証で攻撃可能」と書く
- 自分しか影響を受けない脆弱性を「全ユーザーが影響」と書く

### なぜ致命的か
理論的に不可能な影響を主張すると、トリアージャーから「**専門知識がないとみなされる**」。1 度この烙印を押されると：
- そのレポートは即時 N/A / Informative
- 同じハンターからの後続レポートも先入観を持って読まれる
- Reputation スコアが下がる
- プライベートプログラム招待が来なくなる

### 回避法
- 主張する Impact / Severity が**論理的に成立する経路**を Attacker → Victim → System の 3 段で書けるか、自分で必ず検証
- 書けないなら主張を下げる
- HackerOne / Bugcrowd の Disclosed Reports で類似事例の Severity を確認
- 「自分にとって重要な発見」と「実際の Severity」を切り離す（情緒的バイアス排除）

### レビューでの指摘テンプレ
> Severity の主張が脆弱性の実態と乖離しています：[具体例]。論理的に成立する攻撃経路を Attacker → Victim → System の 3 段で書けない場合、主張を下げてください。過剰な Severity 主張は、トリアージャーから「専門知識がない」と判断されるリスクが高く、Reputation 棄損に直結します。

## 3. 「Critical 病」（CVSS インフレ）

### 症状
- Reflected XSS を「Critical」
- Open Redirect 単体を「High」
- 自己 XSS を「High」
- Information Disclosure（バージョン情報のみ）を「High」

### 回避法
- HackerOne / Bugcrowd の公開 Disclosed Reports で類似 CVSS を見る
- Bugcrowd VRT（Vulnerability Rating Taxonomy）を参照

### 反対の症状（過小評価）
- RCE を「Medium」
- 全データ取り放題の SQLi を「High」
- マルチテナント越境 IDOR を「Medium」

### レビューでの指摘
> CVSS が過大/過小です。`AC:L/PR:N` の根拠と `S:C` の主張に対する境界越えの根拠を具体的に書いてください。`cvss-guide.md` の典型ベクトル例と比較し、X.X (Severity) が実態に合った値です。

## 4. 「他人を使った実害 PoC」

### 症状
- 他人のアカウントの メアドを実際に変更して PoC 化
- 他社員のセッションを乗っ取って撮影
- 本物の顧客 PII をスクショに含める

### なぜダメか
ほぼ全ての BB プログラム規約で禁止。プログラム永久 Ban、法的リスク。

### 回避法
- 自分の 2 アカウント間で再現
- PoC の証拠は最小限（成功レスポンスの抜粋）
- スクショの PII はマスキング

### レビューでの指摘
> PoC スクショに他ユーザーの PII が写っています（規約違反リスク）。マスキングするか、自分の 2 つのテストアカウント間で再現する形に置き換えてください。

---

## 5. 「Steps と Description が混在」

### 症状
- Description セクションに 50 行のリクエストログがベタ貼り
- Steps が「3 行」だけで、検証に必要な情報が散文の Description 側にしかない

### 回避法
- Description：「**何が・どこで・なぜ**」を 1〜3 段落
- Steps：番号付き 1 アクション 1 行、HTTP リクエストはコードブロックで分離
- PoC：完全な payload とレスポンス抜粋

---

## 6. 「スコープを確認しない」

### 症状
- ワイルドカードスコープ `*.example.com` で発見した未知サブドメインの所有権未確認
- マーケサイト（Out-of-scope）で発見
- サードパーティ（CDN、SaaS）が原因のものを対象組織宛に報告
- プログラム規約の「Out-of-scope vulnerability types」に該当する種別（Self-XSS、Rate Limit、SPF/DMARC 等）を提出

### 回避法
- 提出前に Program Brief を**もう一度**読む
- ワイルドカードで見つけたサブドメインは WHOIS / SSL 証明書 / IP レンジで所有権検証
- Out-of-scope vuln types を必ずリスト化

### レビューでの指摘
> 対象アセットがプログラムスコープ内であることを冒頭に明示してください。所有権の根拠（WHOIS / SSL CN / IP ASN）も記載するとトリアージが速くなります。プログラムの「Out-of-scope vulnerability types」に該当しないかも再確認を。

---

## 7. 「重複報告で 5 個に分割」

### 症状
- 同じパラメータの XSS が 15 エンドポイントに存在 → 15 個別レポート
- 同じ脆弱なライブラリで複数サブドメインに脆弱性 → 別レポート

### 回避法
- 根本原因が同じなら **1 レポートに統合**し、エンドポイント一覧を表で示す
- Bugcrowd / HackerOne 共通：分割提出はノイズと判断され Severity Down or 1 件として処理

### レビューでの指摘
> 同じパラメータ・同じ根本原因の問題は 1 レポートに統合してください。本文に対象エンドポイントの表を入れる形を推奨します（Bugcrowd / HackerOne の標準運用）。

---

## 8. 「圧力・煽り・報酬要求」

### 症状
- 「This is critical, you must respond within 24 hours」
- 「Why is this only Medium? Disclosing publicly if you don't increase」
- emoji / 大文字連打

### なぜダメか
プログラム永久 Ban の典型理由。トリアージャーは人間。

### 回避法
- 異議申し立ては事実ベースで「CVSS の `S:C` の根拠は X です」と書く
- 失礼に感じても 24 時間置いてから返信
- Mediation 制度（HackerOne）/ Support リクエスト（Intigriti）を活用

---

## 9. 「ビジュアル過剰 or ゼロ」

### ゼロ症状
- 複雑な攻撃シーケンスなのに動画なし
- XSS なのにスクショなし

### 過剰症状
- 10 分の無編集動画
- 同じスクショを別アングルで 5 枚
- 30 MB の画像を埋め込み

### 回避法
- 動画は 30 秒〜2 分、要点のみ
- スクショは「URL バー + 結果」が 1 枚で完結する構図
- 添付は `xss-poc.mp4`、`idor-200-resp.png` のように名前で内容が分かるファイル名に

---

## 10. 「言語ミスマッチ」

### 症状
- グローバル H1 / Bugcrowd プログラムに日本語レポート
- 機械翻訳のままで主語/動詞の一致が壊れている英文

### 回避法
- レポートは英語が無難（プログラムが日本語明示している場合のみ日本語）
- DeepL → Grammarly → 自分で 1 回読み返し
- 専門用語（CWE 名、OWASP カテゴリ名）は英語のまま使う

### レビューでの指摘
> グローバル H1 プログラムなので英文での提出を推奨します。日本語版は社内共有用に保管し、提出版は英訳してください。最低限、Title / Steps / Impact / Remediation セクションは英文で。

---

## 11. 「Remediation を省略 or 一般論」

### 症状
- 「サニタイズしてください」
- 「最新版にアップデートしてください」
- Remediation セクション自体がない

### 回避法
- 推奨は具体的な API / ライブラリ名（DOMPurify、Bleach、defusedxml 等）
- 副作用警告（既存リクエストへの影響、互換性）も付記
- 開発者が「コミットメッセージにそのまま使える」レベルが理想

### 推奨セキュアライブラリの参考
- HTTP Headers: Helmet.js (Node), secure_headers (Ruby)
- XSS: DOMPurify (JS), Bleach (Python), Ammonia (Rust)
- CSRF: Gorilla CSRF (Go), Django built-in
- Crypto: Google Tink, Themis
- XML: defusedxml (Python)
- SSRF: ssrf_filter (Ruby), ssrf-req-filter (Node)

---

## 12. 「レポートを 1 回も読み直さない」

### 症状
- typo
- 同じパラグラフが 2 箇所
- リンク切れ
- 添付ファイル名と本文中の参照名が不一致

### 回避法
- 提出前に **24 時間寝かせて** から最終チェック
- `markdownlint` / `Grammarly` で機械チェック
- セルフチェックリストを 1 回通す（SKILL.md の出力フォーマット末尾）

## 12.5. プラットフォーム規約違反（Intigriti 公式禁止事項）

Intigriti の公式トリアージスタンダード（v1.3, 2026-01）で**明示的に禁止**されており、検出されると警告・制裁・アカウント停止に直結する行為。多くは HackerOne / Bugcrowd / YesWeHack でも同様に禁止されている。

### A. 48 時間以上の脆弱性の蓄積・保留
- 発見した脆弱性を意図的に保留して圧力をかける
- 別所への売却を待つ
- 競合プログラムでの報告タイミング調整
- → ユーザーが「もう少し溜めてから出す」発言をしたら警告

### B. 第三者の非公開リークデータの購入・利用
- ダークウェブで購入した認証情報
- 漏洩データベースから取得した PII
- → これらを使った PoC は規約違反

### C. 明示的スコープ外での能動的テスト
- ワイルドカードで見つけた未知サブドメインで深掘り
- 「OOS でしたが念のため」と提出
- → スコープ外と分かった時点でテスト中止が原則

### D. 個人情報（PII / 認証情報）の検証
- 「PoC として実顧客の PII を取得しました」
- 他人のセッションを実際に乗っ取って撮影
- 認証情報（メアド、電話番号）の網羅的列挙
- → 自分の 2 つのテストアカウント間で再現するのが正

### E. 同一脆弱性の複数プラットフォーム同時提出
- HackerOne と Bugcrowd 両方に同じ報告
- → 受領プログラムが重複と判断 → 両方で Reputation 低下

### F. 公開予告を圧力に使う（Disclosure Threat）
- 「修正されないなら公開する」「Severity を上げないと開示」
- → 即時アカウント停止リスク

### レビューでの指摘テンプレ
> Intigriti / HackerOne 公式で禁止されている行為が PoC または記述に含まれています：[該当行為]。修正後再提出を推奨します。これが残ったまま提出すると、警告 → 制裁 → アカウント停止に直結する可能性があります。

## 13. 攻撃シナリオが非現実的（Bugcrowd 指摘の「最大の却下理由」）

### 症状
- 技術的事象（アラートが出た、エラーが出た）に集中するあまり、攻撃者・被害者・システムの 3 段が曖昧
- 「攻撃者は機密情報を取得できる」だけで、Attacker / Victim / System が分離されていない
- 自己 XSS なのに「他のユーザーが影響を受ける」と書く
- CSRF だが SameSite Cookie の検討がない
- 配布経路（メール、SNS、広告）が示されず「ユーザーが攻撃を受ける」だけ

### なぜダメか
- Bugcrowd の公式指摘：**初心者の却下理由は「再現できない」よりも「攻撃シナリオが非現実的」**
- セキュリティチームが見ているのは「**そのバグを使って、攻撃者が被害者に対して何を行えるか**」というストーリー

### 回避法
**Attacker → Victim → System の 3 段で書く**：

1. **準備（Attacker）**：攻撃者が何を準備するか（payload、攻撃用 URL、リスナー設置）+ 配布経路
2. **実行（Victim）**：被害者に求められる最低限のアクション + 前提条件（ログイン状態、特定権限）
3. **結果（System）**：直接的な技術的結果 + 最終的なビジネス的損害

詳細は `report-template.md` の「Attack Scenario」セクション。

### レビューでの指摘テンプレ
> 攻撃シナリオが Attacker → Victim → System の 3 段で分離されていません。Bugcrowd が指摘する通り、初心者の却下理由は「再現できない」より「攻撃シナリオが非現実的」が多いです。`report-template.md` の Impact 例を参考に、攻撃者の準備物・被害者の最低限アクション・システムの最終結果を 3 段で書き直してください。

## レビューでこれらの指摘を出すときの口調

- 否定形ではなく **代替案** を必ず提示する。「ダメ」だけだと初心者は困る。
- プロハンターとして「自分が出すならこう書く」というレベルの修正例を示す。
- 教育目的を意識：このレポートだけでなく **次のレポートでも使える原則** を併記する。
- 品質汚染（スキャナ・AI）の指摘は人格攻撃に聞こえやすいので、事実ベース（「検出サインに該当」）で機械的に伝える。
