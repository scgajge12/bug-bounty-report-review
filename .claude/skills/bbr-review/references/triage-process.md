# トリアージプロセスとレポート状態の予測

レビュー時、ユーザーのレポートが**プラットフォームでどう処理されるか**を予測することは、最も価値が高い情報の 1 つ。「Submit-ready」と言われても、実際には Informative / Duplicate / OOS で返されたら意味がない。このリファレンスは Intigriti 公式トリアージスタンダード（2025-26 v1.3）と HackerOne / Bugcrowd の実運用を統合したもの。

レビューの TL;DR には**予測される処理結果**フィールドを必ず出す。

---

## 1. 主要プラットフォームの状態フロー

レポートは提出後、おおむね以下の状態を遷移する：

```
Submitted → New / Triaged → Accepted (Resolved / Bounty issued)
                          → Informative
                          → Duplicate
                          → Out of Scope
                          → Not Applicable / N/A
                          → Spam
                          → Undecided / Need More Info
```

各プラットフォームでラベル名が微妙に違う：

| 状態 | HackerOne | Bugcrowd | Intigriti | YesWeHack |
|---|---|---|---|---|
| 受理・有効 | Triaged → Resolved | Triaged → Resolved | Accepted → Closed | Triaged → Accepted |
| 既知の重複 | Duplicate | Duplicate | Duplicate | Duplicate |
| 価値なし（脆弱性ではない） | N/A (Not Applicable) | Not Applicable | N/A | N/A |
| 仕様 / 軽微な機能バグ | Informative | P5 / Won't fix | Informative | Informative |
| スコープ外 | Out of Scope | Out of Scope | Out of Scope | Out of Scope |
| スパム | Spam | Spam | Spam | Spam |
| 情報不足 | Need More Info | Need More Info | Undecided | Need More Info |

**意識すべきこと：** ハッカー視点で「Bounty に繋がるのは Accepted/Resolved だけ」。それ以外の状態は基本的に**報酬ゼロ**で、繰り返すと Reputation スコアが下がる。

---

## 2. 各状態の判定基準（Intigriti 公式 + 実運用統合）

### Accepted（採用 / Resolved）
- **基準**：本物の脆弱性であり、PoC が機能し、攻撃シナリオが論理的に成立し、スコープ内、未知のもの。
- **このスキルでの予測**：13 観点で 75 点以上 + 報告蓋然性 Confident/Likely + 観点 13（品質汚染）≥ 8

### Informative
- **公式基準（Intigriti）**：「セキュリティ脅威を直ちに構成しない**機能的バグ**」
- **典型例**：
  - 不正なスコープの Cookie だが盗難証明がない
  - 削除後のリソース残存（実害証明なし）
  - バージョン情報の開示のみ
  - スタックトレースの開示のみ
  - 自己 XSS（多くのプログラムで Informative または OOS）
  - SPF / DMARC レコードの欠如のみ
  - Rate Limit Bypass のみ（実害なし）
  - HTTP Security Header の欠如のみ
- **このスキルでの予測**：観点 6（Impact）≤ 5 または観点 7（攻撃シナリオ）≤ 4 → Informative の可能性高
- **回避策**：実害となる攻撃チェーンを構築するか、報告しない

### Duplicate
- **公式基準（Intigriti）**：「同一脆弱性で追加的インパクトなし」「同一根本原因の複数箇所は**最初の 2 報のみ奨励**」
- **典型例**：
  - 既知の S3 Bucket misconfig の同一クラスタ
  - 同一ライブラリの脆弱性が複数サブドメインに展開（最初の 2 報以降は重複扱い）
- **このスキルでの予測**：簡単には予測できないが、以下のシグナルでフラグ：
  - 公開済みの CVE ベース報告
  - 古いライブラリ（jQuery 1.x, Apache 2.4.x など）の典型脆弱性
  - 公開された Disclosed Reports に類似のもの

### Out of Scope (OOS)
- **公式基準**：プログラムスコープに明示的に記載されていない、またはアウトオブスコープリストに該当
- **典型 OOS（プラットフォーム横断で OOS とされやすいもの）**：
  - サードパーティ SaaS（SendGrid、Zendesk、Intercom 等）への報告
  - マーケサイトのみの脆弱性（プログラムが本番アプリのみ対象）
  - プレプロダクション環境の脆弱性
  - DDoS / DoS（明示許可なき限り OOS）
  - Self-XSS（多くのプログラムで OOS）
  - 物理攻撃 / ソーシャルエンジニアリング
  - ベスト プラクティス違反のみ（実害なし）
  - 期限切れ証明書、SSL/TLS 設定不備のみ
  - Click-jacking on non-sensitive page
  - Logout CSRF
  - Email enumeration
  - Open redirect on documentation/marketing pages
- **このスキルでの予測**：観点 1（スコープ）≤ 3 → OOS リスク高

### Not Applicable / N/A
- **基準**：実害がない、または攻撃シナリオが論理破綻している、または脆弱性として認知されない設計判断
- **典型例**：
  - パスワードポリシーが緩い（脆弱性ではなく仕様）
  - 公開ページでの Click-jacking
  - Anti-CSRF が無いが影響が無い GET エンドポイント
  - サーバ側で正常に処理されている入力に対する「XSS の可能性」
- **このスキルでの予測**：観点 7（攻撃シナリオ）≤ 3 → N/A リスク高

### Spam
- **基準**：スキャナ垂れ流し、AI ハルシネーション、低努力レポート、明らかな業務妨害
- **典型例**：
  - Nuclei / Burp Scanner の出力をそのままコピペ
  - AI 生成丸投げ（存在しない CWE / ライブラリ、辻褄の合わない攻撃シナリオ）
  - 1 行レポート「Your site has bugs」
  - 同じ低品質レポートを大量送信
- **このスキルでの予測**：観点 13（品質汚染）≤ 2 → Spam リスク確定
- **影響**：Spam 判定は **Reputation 棄損** に直結。繰り返しでアカウント停止。

### Undecided / Need More Info
- **公式基準（Intigriti）**：「テスト環境のみ」「未使用コード」「ベンダー認知待ち」「権限なし実装」
- **典型例**：
  - 開発用フラグ付き機能で本番未公開
  - 廃止予定のレガシーエンドポイント
  - 第三者ライブラリのバグでベンダー側修正待ち
- **このスキルでの予測**：直接の予測は困難だが、レビュー時に「これは本番で再現可能か？」を必ずユーザーに確認

## 3. 「予測される処理結果」をどう出力するか

レビューの TL;DR で以下のフォーマットを使う：

```markdown
- **予測される処理結果**: Informative の可能性が高い（60%）
  - 攻撃シナリオが Attacker → Victim → System で破綻している（Self-XSS のため Victim = 自身）
  - 影響度が「Cookie 取得可能」止まりで、ビジネス影響に展開できていない
  - 修正後に再評価すれば Accepted に上がる余地あり
```

複数の状態に分岐する可能性がある場合は確率分布で出す：

```markdown
- **予測される処理結果**:
  - Accepted 30%（脆弱性自体は確実、レポート品質次第）
  - Informative 40%（インパクト記述の弱さで降格）
  - Duplicate 20%（公開ライブラリ脆弱性のため既知の可能性）
  - その他 10%
```

予測の根拠は必ず 1-2 行で示す。「なぜそう予測したか」をユーザーに伝えることで、次のレポートで同じパターンを避けられる。

## 4. プラットフォームのトリアージ業務の実態

### Hai Triage（HackerOne）と半自動トリアージ
- 2025 年から HackerOne は **Hai Triage** という半自動システムを運用開始
- AI が**重複検出・低努力レポートのフラグ**を行ってから人間トリアージに渡る
- 表面的なヒューリスティック（テンプレ的、CWE 抜け、PoC 欠如、本文短すぎ・長すぎ）でフィルタされる
- このため、**機械的な品質チェックでも引っかからない品質**が必要

### トリアージャーの 1 日
- 1 トリアージャーが 1 日に **数十〜100件超** のレポートを捌く
- 1 件あたり最初の 30 秒〜2 分で「次に進むか / 差し戻すか / 即却下か」を判断
- 「**最初の 30 秒で再現できそうか**」がレポートの運命を決める
- HTML/Markdown 整形、明確なステップ、コピペ可能な PoC が決定的に重要

### トリアージャーの「赤信号」シグナル
- タイトルが汎用すぎる
- HTTP リクエストがスクショで貼られていてコピペできない
- 認証情報が書かれていない
- Impact が CIA を 3 つ並べる定型文
- Severity が極端（自動的に CVSS インフレ）
- 不可解な英文（AI 生成または機械翻訳）
- 同じハンターから低品質レポートが続いている履歴

これらに 1 つでも該当すると、トリアージャーは**先入観を持って読み始める**。スキルの観点 12 / 13 がここを潰す。

## 5. ハッカー側の禁止行為（Reputation 棄損リスク）

Intigriti 公式トリアージスタンダードから抜粋。これらは**多くの主要プラットフォームに共通する**禁止事項：

### A. 48 時間以上の脆弱性蓄積・保留
- 発見した脆弱性を意図的に保留して圧力をかける、または別所に売却するのは規約違反
- レビュー時、ユーザーが「もう少し溜めてから出す」発言をしたら警告

### B. 第三者の非公開リークデータ購入
- ダークウェブで購入した認証情報を使って侵入し、報告
- これは Code of Conduct 違反、制裁対象

### C. 明示的スコープ外での能動的テスト
- ワイルドカードで見つけた未知サブドメインで深掘りテスト
- 「OOS でしたが念のため報告します」は警告対象

### D. 個人情報（従業員・顧客認証情報）の検証
- 「PoC として実顧客の PII を取得しました」は禁止
- 認証情報のテストも禁止
- レビュー時、PoC で実 PII が取得されていたら**必ず警告**

### E. その他
- 同一脆弱性を複数プラットフォームに同時提出（HackerOne）
- 公開予告を圧力に使う（Disclosure Threat）
- Bounty 額の交渉で脅迫的態度
- AI 生成・スキャナ出力の未検証提出

レビュー時、これらの兆候を発見したら**必ず指摘**する。「観点 12（トーン）または観点 13（品質汚染）でゼロ点」を出すレベル。

## 6. 状態予測の判定マトリクス（簡易版）

レビュー結果から状態予測を出す簡易ルール：

| 条件 | 予測 |
|---|---|
| 観点 13 ≤ 2（品質汚染重度） | **Spam（90%）** |
| 観点 1 ≤ 3 | **OOS（70%）** |
| 観点 13 ≥ 8 + 観点 6 ≤ 5 | **Informative（60%）** |
| 観点 13 ≥ 8 + 観点 7 ≤ 3 | **N/A（50%）** |
| 観点 13 ≥ 8 + 報告蓋然性 Unlikely | **N/A or Informative（70%）** |
| 13観点合計 ≥ 75 + 報告蓋然性 Confident/Likely | **Accepted（80%）** |
| 著名 CVE / 古いライブラリの典型脆弱性 | **Duplicate（40-60%）** |
| ペネトレ環境のテストデータ感 | **Undecided（30%）** |
| 観点 4（再現手順）≤ 6 + 観点 5（PoC）≤ 6 | **Need More Info（30-40%）**：再現詳細不足でトリアージャーが質問返し |
| 報告蓋然性 Likely + 観点 1 が 4-5（スコープ照合曖昧） | **Need More Info（20-30%）**：所有権/スコープ確認 |

複数条件が重なる場合、確率を合算するのではなく**最も影響の大きい条件を優先**する（例：観点 13 が低ければ他がいくら良くても Spam）。

## 7. レビュー時にユーザーへ伝える「アクション」

各状態に応じた次の一手：

| 予測状態 | 推奨アクション |
|---|---|
| **Accepted** | 提出推奨。微修正のみ。 |
| **Informative** | 影響度を再考、攻撃チェーンを構築、または提出取りやめ |
| **Duplicate** | 公開 Disclosed Reports / CVE / Bug-bounty Twitter で類似事例検索。重複確信なら提出しない |
| **OOS** | プログラム規約再読、所有権再確認、対象外なら提出しない |
| **N/A** | 攻撃シナリオを Attacker→Victim→System で再構築。論理破綻なら提出しない |
| **Spam** | 提出禁止。観点 13 の指摘を全て直してから再評価 |
| **Undecided** | 本番再現性を確認、ベンダー側問題なら別ルート |

レビュー出力で「あなたのレポートは X% の確率で Y になります。理由は Z。回避するには W してください」と具体的に書く。
